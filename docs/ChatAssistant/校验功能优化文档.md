# 校验功能优化与流式输出控制

## 文档概述

本文档记录了对 ArchSpec 系统中 AI Agent 校验功能的优化工作，包括：
1. 流式输出控制逻辑优化
2. 校验循环次数控制
3. 前端校验开关 UI 实现
4. 完整的接口打通

**修改日期**: 2026-01-19
**涉及模块**: AI Agent (后端)、Web 前端、API 接口

---

## 一、问题背景

### 1.1 原始问题

在之前的实现中，校验功能存在以下问题：

1. **流式输出问题**：
   - 未开启校验时，generate 节点的结果会实时流式输出 ✓
   - 开启校验后，generate 节点会执行多次（每次校验后重新生成）
   - 所有 generate 的结果都会流式输出到前端，导致用户看到多次生成过程
   - 用户体验不佳：看到内容反复变化，不知道哪个是最终结果

2. **循环控制问题**：
   - Auditor 节点没有最大循环次数限制
   - 可能陷入无限循环，导致响应时间过长
   - 缺少动态 Prompt 引导，LLM 可能过度追求完美

3. **前端控制问题**：
   - 缺少前端开关来控制是否启用校验
   - 用户无法根据需求灵活选择是否校验

---

## 二、解决方案设计

### 2.1 流式输出控制策略

**核心思路**：根据 `enable_audit` 参数动态控制 generate 节点的流式输出

```
未开启校验 (enable_audit=false):
  Researcher → Generate (流式输出) → END

开启校验 (enable_audit=true):
  Researcher → Generate (不流式) → Auditor →
    ├─ pass → END
    └─ revise → Generate (不流式) → Auditor → ...
```

**实现逻辑**：
- 在 API 层 (`workflow_api.py`) 判断 `enable_audit` 状态
- 动态决定 generate 节点是否加入 `user_visible_nodes`
- 未开启校验：generate 流式输出
- 开启校验：generate 不流式输出（等待最终结果）

### 2.2 循环控制策略

**Auditor 循环控制**：
- 新增 `MAX_AUDIT_LOOPS` 配置参数（默认 2 次）
- 在 `AgentState` 中添加 `audit_loop_count` 跟踪循环次数
- 在 `auditor_condition` 中检查是否达到最大循环次数
- 根据循环次数动态调整 Prompt 引导语

**终止条件（按优先级）**：
1. 达到最大循环次数 → 强制退出
2. LLM 判断通过校验（'pass'）→ 正常退出
3. 有工具调用 → 继续循环
4. LLM 要求修改 → 返回 generate 重新生成

### 2.3 前端控制实现

**UI 设计**：
- 在表单面板中新增"高级选项"卡片
- 添加"启用内容校验"开关（el-switch）
- 提供 Tooltip 说明校验功能的作用和影响
- 生成时禁用开关，防止中途修改

```python
    # 1. 硬性限制：达到最大循环次数
```

## 九、问题修复记录

### 9.1 修复：达到最大循环次数时无输出问题（第二次修复）

**问题描述**：
- 当 Auditor 达到 `MAX_AUDIT_LOOPS` 最大循环次数时，`auditor_condition` 直接返回 `END`
- 导致最后一次 Generate 的结果没有输出到前端
- 用户看不到任何生成内容

**根本原因分析**：

以 `MAX_AUDIT_LOOPS = 3` 为例，执行流程如下：

1. Generate #1 (audit_loop_count=0) → Auditor #1 (递增为1) → revise → Generate #2
2. Generate #2 (audit_loop_count=1) → Auditor #2 (递增为2) → revise → Generate #3
3. Generate #3 (audit_loop_count=2) → Auditor #3 (递增为3) → **auditor_condition 检查 (3 >= 3) → 直接返回 END**

**问题所在**：
- 在第3步，Auditor #3 执行完后，`audit_loop_count=3`
- 原来的 `auditor_condition` (L565-570) 检查到 `loop_count >= MAX_AUDIT_LOOPS`，直接返回 `END`
- 这导致跳过了最后一次 Generate 的输出，用户看不到任何内容

**错误的修复尝试**：
第一次修复尝试在 `generate_condition` 和 API 层添加检查，但没有解决根本问题，因为：
- `auditor_condition` 仍然在达到最大次数时直接返回 `END`
- 导致根本不会执行最后一次 Generate

**正确的解决方案**：

**核心修改：修改 `auditor_condition` 的路由逻辑** ([construction_agent.py:L551-586](apps/agent/service/workflow/construction_agent.py#L551-L586))

关键改动：当达到最大循环次数时，**不要直接返回 END**，而是返回 `"generate"` 让它输出最终结果：

**修改前的错误代码**：
```python
def auditor_condition(state: AgentState):
    last_msg = state["messages"][-1]
    loop_count = state.get("audit_loop_count", 0)

    # ❌ 错误：达到最大次数直接返回 END，跳过了最后一次 Generate
    if loop_count >= MAX_AUDIT_LOOPS:
        logger.warning(f"[Auditor] Reached max loops ({MAX_AUDIT_LOOPS}), forcing exit")
        return END  # ❌ 这里导致没有输出！
```

**修改后的正确代码**：
```python
def auditor_condition(state: AgentState):
    """
    Auditor 循环控制逻辑

    终止条件（按优先级）：
    1. LLM 判断通过校验（'pass'）→ 直接结束
    2. 有工具调用则继续
    3. 达到最大循环次数 (MAX_AUDIT_LOOPS) → 返回 generate 输出最终结果
    4. 默认返回 generate（修改后重新生成）
    """
    last_msg = state["messages"][-1]
    loop_count = state.get("audit_loop_count", 0)

    # 1. 检查是否有工具调用
    if last_msg.tool_calls:
        return "auditor_tools"

    # 2. 检查是否通过校验
    content = last_msg.content
    if "pass" in content.lower():
        logger.info("[Auditor] Content passed validation, ending workflow")
        return END

    # 3. ✅ 关键修复：达到最大次数时，返回 generate 而不是 END
    # 这样可以让最后一次 Generate 输出结果
    if loop_count >= MAX_AUDIT_LOOPS:
        logger.warning(
            f"[Auditor] Reached max loops ({MAX_AUDIT_LOOPS}), "
            f"routing to generate for final output"
        )
        return "generate"  # ✅ 返回 generate，让它输出最终结果

    # 4. 需要修改，返回 generate 重新生成
    logger.info(f"[Auditor] Content needs revision (loop {loop_count}/{MAX_AUDIT_LOOPS}), routing to generate")
    return "generate"
```
