## 1. 当前流程概述

### 1.1 整体架构

```
┌─────────────────────────────────────────────────────────────────┐
│                        前端 (Vue.js)                             │
│  ChatAssistant.vue → workflow.ts → SSE EventSource              │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼ POST /agent/workflow/chat/stream
┌─────────────────────────────────────────────────────────────────┐
│                     API 层 (FastAPI)                             │
│  workflow_api.py → event_generator() → SSE StreamingResponse    │
└─────────────────────────────────────────────────────────────────┘
                              │
                              ▼ astream_events
┌─────────────────────────────────────────────────────────────────┐
│                   工作流引擎 (LangGraph)                         │
│  construction_agent.py → StateGraph → 多节点状态机               │
└─────────────────────────────────────────────────────────────────┘
```

### 1.2 工作流节点流程

```
┌──────────┐    generate_doc    ┌────────────┐    ┌──────────────┐
│  Router  │ ─────────────────► │ Researcher │ ──►│ Researcher   │
│  (意图)   │                    │  (检索)     │◄───│   Tools      │
└──────────┘                    └────────────┘    └──────────────┘
     │                                │
     │ general_chat                   ▼ (循环最多3次)
     ▼                          ┌────────────┐
┌──────────────┐                │  Generate  │
│General Agent │                │  (生成)     │
└──────────────┘                └────────────┘
                                      │
                                      ▼ (如果 enable_audit=true)
                                ┌────────────┐    ┌──────────────┐
                                │  Auditor   │ ──►│  Auditor     │
                                │  (校验)     │◄───│   Tools      │
                                └────────────┘    └──────────────┘
                                      │
                                      ▼ (pass/revise)
                                ┌────────────┐
                                │    END     │
                                └────────────┘
```

### 1.3 关键配置参数

| 参数 | 默认值 | 说明 |
|------|--------|------|
| MAX_RESEARCH_LOOPS | 3 | Researcher 最大循环次数 |
| MAX_AUDIT_LOOPS | 1 | Auditor 最大循环次数 |
| MAX_AUDITOR_TOOL_CALLS | 5 | Auditor 工具调用最大次数 |
| MAX_TOTAL_CHARS | 15000 | 上下文最大字符数 |
| MAX_TOOL_MESSAGES | 30 | 保留的 ToolMessage 数量 |
| MIN_CONTENT_THRESHOLD | 1000 | 检索内容最小阈值 |

### 1.4 数据流

**输入参数：**
- `message`: 生成提示词（包含章节标题和特殊要求）
- `template`: 章节模板内容
- `project_info`: 项目概况信息
- `project_id`: 项目 ID
- `enable_audit`: 是否启用校验

**输出事件（SSE）：**
- `session_start`: 会话开始
- `timeline_step`: 步骤状态更新
- `token`: 流式内容输出
- `tool_call`: 工具调用状态
- `step_result`: 步骤结果（Auditor）
- `error`: 错误信息
- `done`: 会话结束

## 2. 流程合理性评估

### 2.1 优点分析

#### ✅ 架构设计
1. **状态机模式**：使用 LangGraph StateGraph 实现清晰的节点流转
2. **流式响应**：SSE 实现实时反馈，用户体验好
3. **两步执行策略**：思考(Streaming) + 行动(Non-Streaming) 分离，解决了 ChatTongyi 流式工具调用的 Bug
4. **循环控制**：多重保护机制防止无限循环

#### ✅ 工具调用
1. **RAG 检索**：案例库 + 规范库双重检索
2. **相关度过滤**：similarity_threshold=0.55 过滤低质量结果
3. **来源追踪**：检索结果包含来源和相关度信息

#### ✅ 质量控制
1. **Auditor 校验**：可选的内容校验机制
2. **占位符处理**：自动检测和替换 XX、__ 等占位符
3. **长度控制**：智能截断防止超出 token 限制

### 2.2 问题与风险

#### ⚠️ 问题 1：Router 节点可能误判
**现状**：Router 使用简单的 LLM 调用进行意图分类
```python
system_msg = SystemMessage(content="""你是一个意图分类器。
如果用户想要生成、编写、创建施工文档或章节，请返回 'generate_doc'。
其他情况请返回 'general_chat'。""")
```
**风险**：
- 边界情况可能误判（如"帮我查一下防火规范"可能被误判为 generate_doc）
- 没有置信度评估

#### ⚠️ 问题 2：Researcher 循环效率
**现状**：最多循环 3 次，每次都会调用 LLM
**风险**：
- 如果第一次检索已经足够，后续循环浪费资源
- 质量评估 `evaluate_retrieval_quality` 逻辑较简单

#### ⚠️ 问题 3：Auditor 单轮校验限制
**现状**：`MAX_AUDIT_LOOPS=1`，只有一次校验机会
**风险**：
- 复杂文档可能需要多轮校验
- 单轮校验可能遗漏问题

#### ⚠️ 问题 4：上下文长度管理
**现状**：硬编码 `MAX_TOTAL_CHARS=15000`
**风险**：
- 不同章节复杂度不同，固定阈值可能不够灵活
- 截断策略可能丢失重要信息

#### ⚠️ 问题 5：错误恢复机制
**现状**：API 调用失败时直接返回"资料收集完毕"
```python
except Exception as e:
    tool_response = AIMessage(content="资料收集完毕")
```
**风险**：
- 静默失败，用户不知道检索失败
- 可能导致生成质量下降

### 2.3 评估结论

| 维度 | 评分 | 说明 |
|------|------|------|
| 架构设计 | ⭐⭐⭐⭐ | 清晰的状态机设计，易于扩展 |
| 流式体验 | ⭐⭐⭐⭐⭐ | SSE 实现完善，前端展示友好 |
| 工具调用 | ⭐⭐⭐⭐ | RAG 检索有效，但缺少重试机制 |
| 质量控制 | ⭐⭐⭐ | Auditor 功能完整但循环次数受限 |
| 错误处理 | ⭐⭐⭐ | 基本错误处理，缺少重试和降级 |
| 可配置性 | ⭐⭐⭐⭐ | 环境变量配置，但缺少运行时调整 |

**总体评价**：当前流程设计合理，核心功能完整，但在错误恢复、质量评估和灵活性方面有改进空间。

## 3. 改进建议

### 3.1 短期优化（低风险，快速见效）

#### 建议 1：增强 Router 意图分类
```python
# 改进方案：增加置信度评估和边界处理
def router_node(state: AgentState):
    messages = state["messages"]

    # 检查是否有明确的模板和项目信息（强信号）
    has_template = bool(state.get("template", "").strip())
    has_project_info = bool(state.get("project_info", "").strip())

    # 如果有模板和项目信息，直接判定为文档生成
    if has_template and has_project_info:
        return {"intent": "generate_doc"}

    # 否则使用 LLM 判断
    system_msg = SystemMessage(content="""...""")
    response = llm.invoke([system_msg] + messages[-3:])
    # ... 原有逻辑
```

#### 建议 2：优化 Researcher 循环控制
```python
# 改进方案：基于检索质量动态决定是否继续
def evaluate_retrieval_quality(state: AgentState) -> dict:
    tool_messages = [m for m in state["messages"] if isinstance(m, ToolMessage)]

    if not tool_messages:
        return {"should_stop": False, "reason": "", "quality_score": 0}

    # 计算质量分数
    total_content = sum(len(m.content) for m in tool_messages)
    non_empty_count = sum(1 for m in tool_messages if "未检索到" not in m.content)

    quality_score = min(100, (total_content / 5000) * 50 + (non_empty_count / len(tool_messages)) * 50)

    # 质量分数 > 70 时提前退出
    if quality_score > 70:
        return {"should_stop": True, "reason": f"检索质量良好 ({quality_score:.0f}分)", "quality_score": quality_score}

    return {"should_stop": False, "reason": "", "quality_score": quality_score}
```

#### 建议 3：增加错误重试机制
```python
# 改进方案：工具调用增加重试
from tenacity import retry, stop_after_attempt, wait_exponential

@retry(stop=stop_after_attempt(3), wait=wait_exponential(multiplier=1, min=2, max=10))
def _retrieve_from_kb_with_retry(query: str, kb_ids: list, kb_name: str) -> str:
    return _retrieve_from_kb(query, kb_ids, kb_name)
```

### 3.2 中期优化（需要测试验证）

#### 建议 4：动态 Auditor 循环
```python
# 改进方案：根据文档复杂度动态调整校验次数
def calculate_audit_loops(template: str, draft_content: str) -> int:
    """根据文档复杂度计算校验次数"""
    # 计算复杂度指标
    table_count = template.count('|')  # 表格数量
    section_count = template.count('##')  # 章节数量
    content_length = len(draft_content)

    complexity_score = (table_count * 2 + section_count + content_length / 1000) / 10

    if complexity_score < 3:
        return 1
    elif complexity_score < 6:
        return 2
    else:
        return 3
```

#### 建议 5：检索结果缓存
```python
# 改进方案：相同查询复用结果
from functools import lru_cache
import hashlib

@lru_cache(maxsize=100)
def cached_retrieve(query_hash: str, kb_ids_tuple: tuple, kb_name: str) -> str:
    return _retrieve_from_kb(query_hash, list(kb_ids_tuple), kb_name)

def retrieve_with_cache(query: str, kb_ids: list, kb_name: str) -> str:
    query_hash = hashlib.md5(query.encode()).hexdigest()
    return cached_retrieve(query_hash, tuple(kb_ids), kb_name)
```

### 3.3 长期优化（架构级改进）

#### 建议 6：引入质量评估节点
```
Router → Researcher → Generate → QualityEvaluator → Auditor/END
                                        │
                                        ▼
                              (评估生成质量，决定是否需要校验)
```

#### 建议 7：支持增量生成
- 对于长文档，支持分段生成
- 每段生成后可以独立校验
- 减少单次 token 消耗

```python
import httpx
from typing import AsyncGenerator
```

### C. SSE 事件协议

```typescript
// 事件类型定义
interface SSEEvent {
  event: 'session_start' | 'timeline_step' | 'token' | 'tool_call' | 'step_result' | 'error' | 'done';
  data: {
    // session_start
    timestamp?: number;

    // timeline_step
    id?: string;
    title?: string;
    status?: 'in_progress' | 'completed' | 'failed';
    node?: string;
    duration?: number;
    thought?: string;

    // token
    content?: string;

    // tool_call
    step_id?: string;
    tool_id?: string;
    name?: string;
    display_name?: string;
```
